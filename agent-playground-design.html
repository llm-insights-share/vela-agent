第1章AGENT MANAGEMENT OVERVIEW

Agent 管理概览

Agent Playground 是 Vela 平台中面向 Agent 开发者和管理员的核心工作台，提供 Agent 全生命周期管理能力——从创建、配置、验证调试到发布上线，以及模型服务的基础设施管理。

CORE

Agent 生命周期管理

覆盖 Agent 的创建、修改、下架、删除全生命周期。每个 Agent 拥有独立版本号，支持版本化迭代和回滚。

  • 新建 Agent：配置名称、描述、主模型
  • 修改 Agent：更新配置，自动递增版本号
  • 下架 Agent：保留数据，暂停服务
  • 删除 Agent：彻底清除，不可恢复
QUALITY

验证与调试

提供配置校验、沙盒测试和实时调试工具链，确保 Agent 在发布前经过充分验证。

  • 配置合规性自动校验
  • 沙盒环境隔离测试
  • 推理过程可视化调试
  • 测试用例管理与回归
INFRA

模型服务管理

统一管理模型供应商和模型服务，支持主流大模型供应商接入，灵活配置模型端点。

  • 供应商：OpenAI、Anthropic、百炼、DeepSeek
  • 支持自定义 base_url 接入
  • 模型列表实时查询
  • 模型服务增删改管理

设计原则：Agent Playground 遵循"配置即代码、版本即快照、测试即保障"的设计理念。Agent 的每一次变更都产生新的版本记录，确保可追溯、可回滚；模型服务与 Agent 解耦，Agent 通过引用模型服务 ID 来绑定推理引擎，模型服务的变更不影响已发布的 Agent 版本。

第2章AGENT CREATION

Agent 创建

Agent 创建是 Agent Playground 的核心入口流程。用户通过表单配置 Agent 的基本信息、选择主模型，系统自动生成初始版本号并完成 Agent 实例化。

2.1 基本信息与版本号

Agent 基本信息字段

字段名类型必填说明
namestring✅Agent 名称，全局唯一，2-50 字符
descriptiontext✅Agent 功能描述，10-500 字符
model_service_idUUID✅引用的模型服务 ID（从供应商模型列表中选择）
system_prompttext✅系统提示词，定义 Agent 角色和行为
dept_idUUID✅所属部门/业务单位 ID
autonomy_levelenum✅自主度等级：L1（需HITL）/ L2 / L3（全自动）
max_concurrent_sessionsint✅最大并发会话数，默认 5
token_budgetint✅单次会话 Token 预算上限，默认 8000
skill_pack_idsUUID[]否绑定的 Skill 包 ID 列表
knowledge_base_idsUUID[]否关联的知识库 ID 列表
tool_permissionsjson否工具权限映射，如 {"erp_query":"allowed","code_execute":"sandboxed"}
tagsstring[]否标签，用于分类和检索

版本号策略

每个 Agent 从创建起即带有版本号，遵循 SemVer 2.0 规范（MAJOR.MINOR.PATCH），版本号随 Agent 修改自动递增。

版本位递增条件示例
MAJOR主模型更换、自主度级别变更、不兼容的 Skill 包升级v1.0.0 → v2.0.0
MINOR新增 Skill 包、新增知识库、新增功能配置v1.0.0 → v1.1.0
PATCHsystem_prompt 微调、token_budget 调整、描述更新v1.0.0 → v1.0.1

版本不可变：已发布的版本内容不可修改。任何对 Agent 配置的修改都将创建一个新的版本记录。历史版本保留在数据库中，支持查看、对比和回滚。

版本记录结构

PythonAgentVersion 核心字段
class AgentVersion(BaseModel):
    version_id: UUID          # 版本唯一标识
    agent_id: str            # 所属 Agent ID
    version: str             # 语义版本号，如 "1.2.0"
    version_seq: int         # 递增序号，从 1 开始
    change_type: ChangeType   # MAJOR|MINOR|PATCH
    change_summary: str     # 变更摘要
    snapshot: dict           # 该版本的完整配置快照
    status: VersionStatus    # DRAFT|TESTING|PUBLISHED|DEPRECATED
    created_by: str          # 创建者 user_id
    created_at: datetime

2.4 工具权限配置

Agent 创建时可配置工具权限映射（tool_permissions），精细化控制 Agent 在推理过程中对各项工具/技能的调用权限。这是保障 Agent 安全运行的关键配置，遵循最小权限原则。

权限级别说明

权限级别含义典型场景
allowed允许调用，工具直接执行数据分析、文档生成等只读操作
sandboxed沙盒执行，在隔离容器中运行代码执行、文件操作等可能产生副作用的行为
denied拒绝调用，即使 Skill 包已绑定敏感数据访问、跨部门操作等越权行为
ask_first执行前需征求用户确认发送通知、修改数据等需要人工决策的操作

设计原则：tool_permissions 是 Agent 配置的最终控制层。即使 Skill 包提供了工具实现，如果 tool_permissions 中设置为 denied，Agent 在推理时将无法调用该工具。这是防止 Agent 越权操作的安全阀门。

权限配置示例

JSONtool_permissions 配置示例
{
  "tool_permissions": {
    "erp_query": "allowed",
    "erp_update": "ask_first",
    "code_execute": "sandboxed",
    "doc_generate": "allowed",
    "email_send": "ask_first",
    "hr_data_access": "denied",
    "finance_approve": "denied",
    "calendar_schedule": "ask_first"
  }
}

权限继承规则

场景继承规则优先级
Skill 包未绑定该 Skill 下的所有工具自动 denied—
Skill 包已绑定，tool_permissions 未配置继承 Skill 包的默认权限低
Skill 包已绑定，tool_permissions 已配置tool_permissions 覆盖 Skill 包默认值高
自主度 L1 级别强制启用 ask_first 安全策略最高
第3章AGENT MODIFICATION

Agent 修改

Agent 修改操作会创建新的版本记录，根据变更内容自动决定版本号递增策略。修改操作支持对已发布 Agent 的在线配置更新，也可对草稿状态的 Agent 继续编辑。

修改规则

修改内容版本递增是否需要重新验证是否影响线上 Agent
描述、标签、配额PATCH否立即生效（热更新）
system_promptPATCH建议立即生效（热更新）
新增 Skill 包 / 知识库MINOR是需要重新发布
删除 Skill 包 / 知识库MINOR是需要重新发布
更换主模型MAJOR是（强制）需要审核 + 重新发布
修改自主度等级MAJOR是（强制）需要审核 + 重新发布

修改约束：已发布（PUBLISHED）的 Agent 在修改时，线上运行的会话继续使用当前版本直到会话结束。新会话将使用修改后的版本（如果是热更新）或等待新版本发布后再切换。

修改流程

STEP 1

进入编辑模式

在 Agent 详情页点击"修改"，进入编辑页面。表单预填当前版本配置，锁定不可修改的版本号字段

STEP 2

修改配置

用户修改需要变更的字段。系统实时计算本次变更将触发的版本递增类型（MAJOR/MINOR/PATCH）并展示预览版本号

STEP 3

变更对比

系统展示修改前后 Diff 对比视图，帮助用户确认变更内容。用户填写 change_summary 说明变更原因

STEP 4

提交修改

如果是热更新变更（PATCH），立即更新数据库并通知 M3 Agent 管理模块刷新缓存。如果涉及 MAJOR/MINOR，创建新 DRAFT 版本

STEP 5

版本记录

系统生成新版本记录，快照完整配置，写入 agent_versions 表。旧版本保留不变，可随时查看和回滚

修改请求示例

JSONPUT /api/v1/agents/{agent_id}
{
  "description": "自动分析采购订单数据，识别异常采购模式，生成风险报告（支持多维度分析）",
  "token_budget": 12000,
  "change_summary": "增加Token预算以支持更复杂的多维度分析场景"
}
// 仅修改描述和配额 → PATCH 递增 → v0.1.0-draft → v0.1.1-draft
第4章DEPRECATION & DELETION

Agent 下架与删除

Agent 下架是一个可逆的停服操作，保留所有数据和版本历史；Agent 删除则是不可逆的彻底清除操作，需要满足前置条件并通过二次确认。

下架（Deprecate）

属性说明
操作性质可逆，Agent 状态从 PUBLISHED → DEPRECATED
前置条件无活跃会话（所有会话已关闭或等待关闭）
行为拒绝新会话建立，已有会话可继续完成或等待超时
数据保留全部配置、版本历史、会话记录完整保留
恢复可重新发布恢复服务（状态回 PUBLISHED）
权限需要部门管理员审批
通知下架后邮件通知该 Agent 的活跃用户

安全下架策略：下架请求提交后，Agent 首先进入 DEPRECATING 过渡状态，拒绝新会话但允许现有会话完成（最长等待 30 分钟），之后自动转入 DEPRECATED 状态。

删除（Delete）

属性说明
操作性质不可逆，Agent 及其关联数据被物理删除
前置条件Agent 必须处于 DEPRECATED 状态；无任何活跃或历史会话引用
行为删除 Agent 记录、所有版本快照、会话历史、会话消息
数据保留记忆模块（M7）中的提炼经验可选择保留或级联删除
恢复不可恢复，需重新创建
权限需要平台管理员审批 + 二次确认
审计删除操作写入不可篡改的审计日志

删除保护：Agent 创建后 7 天内不可删除（保护期）。删除请求需提供删除原因，经过平台管理员审批（T+24h）后方可执行，期间可随时取消删除请求。

下架与删除流程对比

状态流转: PUBLISHED ──(下架申请)──→ DEPRECATING ──(会话清空)──→ DEPRECATED │ │ │ (等待7天保护期) │ │ │ (删除申请 + 审批) │ │ └────────────────(永不删除)────────────────────────────│ ▼ DELETED (物理删除)
第5章VALIDATION & DEBUGGING

Agent 验证和调试

Agent Playground 提供完整的验证和调试工具链，覆盖配置校验、沙盒测试和实时推理调试三个阶段，确保 Agent 在发布前经过充分的质量验证。

5.1 配置校验

配置校验在 Agent 创建和每次修改后自动触发，确保所有配置项的合法性、一致性和兼容性。

校验类别校验项严重级别
名称校验全局唯一性、长度 2-50、允许字符集ERROR
模型可用性model_service_id 存在且 status=activeERROR
配额合法性max_concurrent_sessions ≥ 1, token_budget ≥ 1000ERROR
Skill 兼容性Skill 包版本与 Agent 类型兼容ERROR
知识库可访问知识库存在且 Agent 有读取权限WARN
system_prompt非空、长度 ≤ 32000、不含注入风险WARN
自主度与模型匹配L1 级别须配置 HITL 审批通道WARN

校验结果展示：校验结果以清单形式展示，ERROR 级别的问题必须修复后 Agent 才能进入测试阶段；WARN 级别可跳过但建议修复。每个问题附带修复建议链接。

校验结果示例

{
  "passed": false,
  "errors": [],
  "warnings": [
    {
      "field": "system_prompt",
      "message": "system_prompt 建议增加安全护栏声明",

      "severity": "WARN"
    }

}

5.2 沙盒测试

配置校验通过后，Agent 进入沙盒测试环境。沙盒环境是一个与生产环境完全隔离的独立实例，使用相同的配置但受限的资源配额，确保测试不影响线上服务。

沙盒环境特性

特性沙盒环境生产环境
资源配置最小配额（1并发、2k Token）配置的完整配额
数据访问测试数据 / 只读副本真实业务数据
工具调用Mock 模式（可选真实调用）真实工具调用
外部依赖全部隔离，不产生副作用真实 API 调用
会话记录独立存储，标注 sandbox=true生产存储
计费不计入部门配额计入部门配额

测试模式

MODE 1
交互式测试（Chat Debug）
开发者通过 Chat UI 与沙盒 Agent 进行对话交互，实时观察 Agent 的推理过程、工具调用和输出结果。支持查看每一步的 ReAct 思考链。
MODE 2
自动化测试（Test Suite）
开发者预设测试用例（输入 + 期望输出或评估标准），系统自动批量执行并生成测试报告。支持回归测试，每次修改后自动重跑。
MODE 3
A/B 对比测试
同时运行当前版本和新版本的沙盒实例，对比同一输入下的输出差异。用于评估 Prompt 优化、模型切换的效果。

5.3 调试工具

TRACE

推理过程可视化

实时展示 Agent 的ReAct 推理链，包括 Thought（思考）、Action（动作）、Observation（观察）每一步的详细内容。支持时间轴视图和树状展开。

  • 每步推理的 Token 消耗和时间消耗
  • 工具调用的请求/响应完整记录
  • 异常步骤高亮标红

第6章AGENT PUBLISHING

Agent 发布

Agent 发布是将经过验证的 Agent 版本从测试环境推送到生产环境的过程。发布流程包含审核、灰度发布和全量上线三个阶段，确保 Agent 的稳定性和安全性。

6.1 发布流程

STEP 1

提交发布申请

开发者确认测试通过后，在 Agent 详情页点击"提交发布"，填写发布说明（release_notes），指定目标版本

STEP 2

自动化检查

系统自动执行发布前检查：配置校验全部通过、测试用例通过率 ≥ 95%、无高危安全告警、沙盒测试无异常

STEP 3

部门管理员审核

审核发布申请，检查 Agent 能力范围、数据访问权限是否符合最小权限原则。审核不通过则驳回并附原因

STEP 4

灰度发布

审核通过后，Agent 版本状态变更为 PUBLISHING。系统将 10% 的新会话流量路由到新版本，观察 1 小时

STEP 5

全量上线

灰度期间无异常（错误率 < 1%、P95延迟无劣化），自动切换为 100% 流量。版本状态变更为 PUBLISHED

STEP 6

旧版本处理

旧版本状态变更为 DEPRECATED。已建立的会话继续使用旧版本直到会话结束，新会话使用新版本

发布前检查清单

检查项通过条件阻塞发布
配置校验0 ERROR✅ 是
测试用例通过率≥ 95%✅ 是
安全扫描无高危漏洞✅ 是
模型健康检查模型服务 status=active✅ 是
Skill 包兼容性版本号匹配✅ 是
配额评估部门月度 Token 余量 ≥ 20%否（仅告警）
文档完整性Agent 描述非空否（仅告警）

发布请求示例

JSONPOST /api/v1/agents/{agent_id}/publish
{
  "version_id": "ver_01J2XK...",
  "release_notes": "v1.2.0: 新增供应商图谱分析 Skill，优化异常检测 Prompt，Token 预算提升至 12000",
  "strategy": "canary",
  "canary_percentage": 10,
  "canary_duration_minutes": 60,
  "auto_rollback_on_error": true,
  "error_threshold_pct": 5
}

6.2 版本发布策略

STRATEGY 1

蓝绿部署

适用于MAJOR 版本升级，新旧版本同时运行，通过路由器一键切换流量。切换后发现异常可立即回滚至旧版本。

STRATEGY 2

金丝雀发布

适用于MINOR 版本升级，逐步将新版本流量从 10% → 50% → 100%，每阶段观察 10 分钟，异常自动回滚。

STRATEGY 3

热更新

适用于PATCH 版本升级（描述、配额等），立即更新生产配置，M3 Agent 管理模块刷新缓存后即时生效。

回滚机制：所有发布策略均支持一键回滚。回滚操作将版本状态恢复为 PUBLISHED（回滚目标版本），并将当前版本标记为 DEPRECATED。回滚记录写入版本历史和审计日志，可追溯。

8.1 Skill包管理

Skill 包是 Agent 可调用的技能集合，每个 Skill 包包含若干工具（Tool）。Skill 包按作用域分为三层：平台通用 Skill、业务类型 Skill、部门专属 Skill，Agent 可通过绑定 Skill 包来扩展其能力。

第10章API SPECIFICATION

接口规范

Agent Playground 对外暴露 RESTful API，覆盖 Agent 管理、模型服务管理和系统管理三大领域。

Agent 管理接口

GET/api/v1/agents
查询 Agent 列表。支持按 name、status、dept_id、tags 筛选，支持分页和排序。返回 Agent 基本信息（含当前版本号和模型名称）。
POST/api/v1/agents
创建新 Agent。请求体包含 name、description、model_service_id、system_prompt、dept_id 等字段。校验通过后生成初始草稿版本 v0.1.0-draft。
GET/api/v1/agents/{agent_id}
查询 Agent 详情，包含当前版本配置、版本历史列表、关联的 Skill 包和知识库、发布记录等完整信息。
PUT/api/v1/agents/{agent_id}
修改 Agent 配置。根据修改内容自动决定版本递增类型（MAJOR/MINOR/PATCH），生成新的版本快照。
DELETE/api/v1/agents/{agent_id}
删除 Agent。需满足前置条件（状态为 DEPRECATED、无活跃会话），需平台管理员审批和二次确认。

版本与发布接口

GET/api/v1/agents/{agent_id}/versions
查询 Agent 的全部版本历史，按 version_seq 降序排列。每个版本包含 change_type、change_summary、status 等字段。
GET/api/v1/agents/{agent_id}/versions/{version_id}/diff
对比两个版本的配置差异（当前版本 vs 上一个 PUBLISHED 版本），以结构化 Diff 格式返回变更内容。
POST/api/v1/agents/{agent_id}/validate
触发配置校验，返回校验结果清单（errors/warnings）。Agent 修改后可手动触发重新校验。
POST/api/v1/agents/{agent_id}/publish
提交发布申请。指定 version_id、发布策略（blue-green/canary/hot-update）、灰度比例等参数。触发审核流程。
POST/api/v1/agents/{agent_id}/rollback
回滚到指定版本。将目标版本状态恢复为 PUBLISHED，当前版本标记为 DEPRECATED。需管理员权限。

模型服务管理接口

GET/api/v1/model-services
查询模型服务列表。支持按 provider、status、capabilities 筛选。按供应商分组展示，标记健康状态。
POST/api/v1/model-services
手动新增模型服务。需填写 provider_id、model_name、display_name、max_tokens 等字段。保存后自动执行健康检查。
PUT/api/v1/model-services/{id}
修改模型服务信息。可修改 display_name、capabilities、max_tokens。不可修改 provider 和 model_name。
DELETE/api/v1/model-services/{id}
删除模型服务。需检查无 Agent 引用、无活跃会话。返回级联影响清单或确认删除。
POST/api/v1/providers/{id}/sync-models
从供应商 API 实时同步模型列表到本地 model_services 表。增量更新，新增的模型自动创建，已存在的模型更新信息。
第11章STATE MACHINE & LIFECYCLE

状态机与生命周期

定义 Agent 实例、Agent 版本和模型服务的状态机及其生命周期流转规则。

Agent 实例状态机

DRAFT ──→ VALIDATING ──→ TESTING ──→ PUBLISHED │ │ │ │ │ │ │ ├──→ DEPRECATING → DEPRECATED │ │ │ │ │ │ │ │ └──→ SUSPENDED（违规/紧急） │ │ │ │ │ │ │ (可恢复或删除) │ │ │ │ │ └──→ TEST_FAILED（测试未通过） │ │ │ │ │ (修改后重回 VALIDATING) │ │ │ └──→ VALIDATION_FAILED（校验失败） │ │ │ (修改后重回 DRAFT) │ └──(直接删除，不可恢复) 各状态含义： DRAFT 草稿：刚创建，可自由修改 VALIDATING 校验中：正在执行自动化配置检查 VALIDATION_FAILED 校验失败：存在 ERROR 级别问题 TESTING 测试中：沙盒环境运行测试用例 TEST_FAILED 测试失败：测试用例通过率 < 95% PUBLISHED 已发布：生产环境运行，接受会话请求 SUSPENDED 已暂停：违规或紧急情况暂停服务 DEPRECATING 下架中：拒绝新会话，等待现有会话结束 DEPRECATED 已下架：不接受任何会话，数据保留 DELETED 已删除：物理删除（不可逆）

Agent 版本状态机

DRAFT ──→ TESTING ──→ PUBLISHING ──→ PUBLISHED │ │ │ │ │ │ │ ├──→ DEPRECATED │ │ │ │ │ │ │ │ │ (被新版本替代) │ │ │ │ │ │ └──→ ROLLBACK │ │ │ (发布失败回滚) │ │ │ └──→ TEST_FAILED │ └──(废弃，永不发布)

模型服务状态机

ACTIVE ←────────── INACTIVE ──────────→ DELETED │ ↑ │ ↑ │ │ (恢复) │ (停用) │ (级联检查) │ └───────────────────┘ │ │ │ └──→ ERROR ──(修复)──→ ACTIVE │ │ │ └──(持续异常)──→ INACTIVE │ (管理员手动停用) 模型服务被 Agent 引用且 Agent 状态为 PUBLISHED 时： → 不允许删除模型服务 → 不允许将模型服务状态改为 INACTIVE（仅允许 ERROR）
