第1章DEFINITION & POSITIONING

业务智能体定义与定位

业务智能体（Business Agent）是 Vela 平台中面向具体业务场景的智能执行单元，能够独立接受、处理来自人类员工或其他智能体的会话请求，并在限定资源配额下并发服务多个会话。

1.1 核心概念与定位

IDENTITY

什么是业务智能体

业务智能体是一个有状态的、具备特定业务专长的 AI 执行体。它不是单次函数调用，而是一个可被"雇用"的持续在线服务——拥有自己的技能包、知识库、经验沉淀和中长期记忆。

  • 可被员工或其他 Agent 发起会话请求
  • 在一次会话中维持完整的上下文状态
  • 可同时处理多个独立会话（受配额约束）
SCOPE

业务智能体 vs 通用智能体

业务智能体是通用智能体框架的业务专化实例。同类业务智能体共享相同的能力基础，但因所属业务单位不同而在技能细节、知识范围、业务数据接入上存在差异。

  • 同类型 Agent 在不同业务单位有独立实例
  • 能力（Skill）大部分共享，部分业务专化
  • 知识库与业务数据完全按业务单位隔离
VALUE

核心价值主张

业务智能体通过能力共享 + 数据隔离的架构，实现了"一套框架服务全企业"的规模效益，同时保证各业务单位的专业深度和数据安全。

  • 降低 Agent 开发与维护成本
  • 各业务单位知识和数据严格隔离
  • 技能迭代一处升级、全部受益

设计原则：一个业务智能体的 Skill（技能）、知识、经验、中长期记忆 在其所有并发会话之间共享；不同业务单位的同类业务智能体之间，技能大体相同（可能有微差），但知识库与业务数据完全独立。

第2章SESSION MODEL

会话模型设计

业务智能体采用多会话并发模型：同一个 Agent 实例可以同时承载多个来源不同的会话；每个会话是独立的执行上下文，同时可以以受控方式引用其他会话的历史信息。

2.1 会话建立与隔离

会话发起方类型

发起方类型标识协议典型场景
内部员工user_id (JWT)WebSocket / HTTP员工在 Chat UI 向 Agent 提问
其他业务智能体agent_id (A2A)A2A /tasks审计 Agent 委派财务 Agent 分析数据

会话建立流程

STEP 1

身份验证

网关 M1 完成 JWT / API Key / Agent Card 认证，提取 caller 身份和权限

STEP 2

配额预检

检查 Agent 当前并发会话数是否达到配额上限；若已满，返回 503 并附带 retry_after 提示

STEP 3

会话分配

Agent 管理模块 M3 分配 session_id（UUID v7，含时间戳），写入 PostgreSQL sessions 表，并在 Redis 创建热缓存条目

STEP 4

上下文初始化

记忆模块 M7 按会话作用域加载：企业级记忆（全局）、部门级记忆（该 Agent 所属部门）、个人级记忆（若 caller 是员工）

STEP 5

推理引擎就绪

M4 推理执行引擎为该会话初始化 ReAct 状态机，注入 system_prompt + 上下文记忆，建立 WebSocket 流或 HTTP 长轮询通道

会话隔离保证

会话级隔离：每个会话拥有独立的 session_id、独立的消息历史（messages 表）、独立的任务执行状态（tasks 表）。不同会话的推理过程互不感知，不会相互干扰。

资源隔离：每个会话有独立的 Token Budget（来自 AgentConfig.token_budget）；单次推理超预算会触发降级而非影响其他会话。会话级 TraceID 确保全链路可观测，互不混淆。

2.2 会话上下文的跨会话参考机制

设计约束：不同会话的当前执行上下文（消息流、任务状态）互相隔离，不可直接读写。但经过提炼的历史经验与记忆（已写入记忆模块 M7）可被同一 Agent 的所有会话参考，实现跨会话知识积累。

可参考的跨会话信息

信息类型存储位置参考粒度访问控制

部门经验总结M7 部门级记忆同部门会话可读部门只读，审批写

跨会话参考调用示意

设计记忆检索 — 跨会话积累
# 会话 A 结束时，Mem0 自动提炼经验
async def on_session_close(session: Session):
    insights = await mem0.extract(
        messages=session.messages,
        min_importance=0.7
    )
    # 写入个人级记忆，供未来会话参考
    await memory_store.save(insights,
        scope="personal",
        caller_id=session.caller_id
    )

# 会话 B 启动时，加载历史积累
async def on_session_init(session: Session):
    memories = await memory_store.retrieve(
        agent_id=session.agent_id,
        caller_id=session.caller_id,
        scopes=["enterprise","dept","personal"]
    )
    # 注入到会话上下文，供推理参考
    session.context_snapshot["memories"] = memories

2.3 并发会话数控制

业务智能体能同时处理的会话数量，由其可使用资源配额决定。配额从两个维度约束：计算资源（CPU/内存/GPU Token 消耗速率）和逻辑并发上限（max_concurrent_sessions）。两者均满足时，新会话才被接受。

配额维度约束方式超限行为
max_concurrent_sessionsRedis 原子计数器503 + retry_after
token_rate_limit滑动窗口 (tokens/min)排队等待 or 降级
dept_token_monthlyRedis 月度累计降级到小模型
single_session_budgetper-session 计数会话内降级/截断

弹性设计：max_concurrent_sessions 是软上限，可由平台管理员通过 M10 系统管理模块动态调整，无需重启 Agent 实例。当资源扩容后，上限自动生效。

并发控制实现

Python会话准入控制 — Redis 原子操作
async def try_acquire_session_slot(
    agent_id: str,
    session_id: str
) -> bool:
    key = f"agent:sessions:{agent_id}"
    quota = await get_quota(agent_id)
    
    # Lua 脚本原子操作，避免竞态
    result = await redis.eval("""
      local cur = redis.call('SCARD', KEYS[1])
      if cur >= tonumber(ARGV[1]) then
        return 0
      end
      redis.call('SADD', KEYS[1], ARGV[2])
      return 1
    """,
        keys=[key],
        args=[quota.max_concurrent, session_id]
    )
    return bool(result)
第3章SHARED CAPABILITY LAYER

共享能力层设计

同一业务智能体的 Skill（技能）、知识、经验与中长期记忆，在该 Agent 的所有并发会话之间共享，但各自有不同的读写权限和更新机制。

⚡
Skill · 技能包
MCP 工具调用能力、ReAct 推理策略、业务方法论
全会话只读共享
📚
Knowledge · 知识库
向量化文档、知识图谱、FAQ 库
全会话只读共享
🧪
Experience · 经验库
历史案例、成功模式、失败教训（部门级记忆沉淀）
全会话读 / 审批写
🧠
Long-term Memory · 中长期记忆
三层记忆（企业/部门/个人），Mem0 自动提炼
分层读写控制

关键设计原则：共享并不意味着所有会话能同时写入。Skill 和知识库是只读引用（版本化管理，更新走发布流程）；经验和记忆的更新走异步提炼 + 审批沉淀路径，保证读时一致性，写时无争用。

业务智能体实例 (AGENT-AUD-01) ├── [只读共享] Skill 包 v2.3.1 │ ├── 审计查证方法生成 │ ├── 统计异常检测 │ └── MCP: ERP查询 / 文档生成 │ ├── [只读共享] 知识库 (审计部) │ ├── Milvus: kb_aud_* collections │ └── Neo4j: audit_* graph │ ├── [会话完成后异步写] 经验沉淀 │ └── M7 部门级记忆 ← Mem0 提炼 │ └── [各会话持有] 个人级记忆 └── 随会话 caller_id 载入

3.1 Skill 技能体系（共享 + 版本化）

Skill 是业务智能体的能力合集，以版本化的 SkillPack 形式存储于 M6 技能与工具模块，通过 MCP 协议调用。一个 SkillPack 可绑定多个 Agent，所有绑定 Agent 的全部会话共同调用同一版本的 Skill 实现。

Skill 分类共享范围更新机制
平台通用 Skill所有 Agent 共享平台统一发布
业务类型 Skill同类 Agent 共享业务线审核发布
业务单位专化 Skill单部门 Agent部门开发者发布

Skill 调用隔离机制

虽然 Skill 实现共享，但每次调用是独立的无状态请求。MCP Server 的沙箱容器为每次 tool_call 独立启动，天然隔离不同会话的代码执行上下文。

会话A: tool_call → 技能路由器 会话B: tool_call → 技能路由器 会话C: tool_call → 技能路由器 ↓ [共享] 同一 MCP Server 端点 ↓ 每次调用独立 Docker 容器实例 (network=none, memory=256MB, timeout=30s) ↓ 结果返回各自会话，互不可见

无状态工具原则：所有 MCP 工具严格设计为无状态，工具执行结果完全由输入参数决定，不依赖任何会话外部状态，从而保证多会话并发调用的正确性。

3.2 知识与经验共享

知识库（Knowledge）— 静态共享

知识库是 Agent 的专业背景知识，通过 M5 知识库管理模块向量化存储，所有会话通过 RAG 检索共享同一知识库版本。知识库更新走独立的摄取流水线，不影响正在进行的会话。

属性设计决策
读并发无限制，Milvus HNSW 索引支持高并发读
写更新走摄取流水线，新版本灰度切换，不中断会话
一致性会话内读同一快照版本；跨会话可能读到不同版本
隔离按 dept_id 分 Milvus Collection，物理隔离

经验库（Experience）— 动态沉淀

经验是从已完成会话中提炼的高价值知识，写入 M7 部门级记忆，供该 Agent 未来所有会话参考。更新需通过 Mem0 提炼 + 审批的两阶段流程，确保质量。

T+0

会话完成

会话关闭时发布 session.closed 事件到 Kafka

T+30s

Mem0 提炼

异步从消息历史中提取重要性 ≥ 0.7 的洞察

T+1min

个人记忆写入

自动写入 caller 的个人级记忆，立即对其下次会话生效

申请

经验上沉申请

员工或 Agent 可申请将个人经验上沉为部门级，全部门共享

审批

部门管理员审核

48h 内审批；通过后复制到部门级记忆，所有后续会话可读

3.3 中长期记忆（共享层次模型）

记忆层共享范围存储TTL写入条件更新频率
企业级记忆该 Agent 所有会话Redis + PostgreSQL永久平台管理员操作极低（战略级信息）
部门级记忆同部门 Agent 的所有会话Milvus + PostgreSQL1 年+部门管理员审批低（月度沉淀）
个人级记忆同一 caller 发起的所有会话Milvus + PostgreSQL1 年Mem0 自动提炼高（每次会话后）

记忆读取时机：会话初始化时，M7 记忆模块按三层并行检索，将相关记忆注入到会话的 context_snapshot 中。推理过程中如需召回新记忆，通过 memory_search 工具显式调用，结果仅对当前会话可见。

第4章QUOTA & CONCURRENCY

配额与并发管理

业务智能体能同时支持的会话数量，由多层配额体系共同决定。配额从平台 → 部门 → Agent → 会话逐层约束，确保资源公平分配。

四层配额体系

层级配额项控制方实现
平台级总并发会话上限、全局 Token/min平台管理员Redis 全局计数
部门级月度 Token 预算、部门并发上限部门管理员Redis 月度滑窗
Agent 级max_concurrent_sessions、token_rateAgent 配置Redis SADD 集合
会话级token_budget（单次任务上限）AgentConfigper-session 计数

配额决定并发能力

资源配额 → 最大并发会话数 的映射示例： 资源配置 token_rate max_concurrent ───────────────────────────────────────── 轻量型 2k tokens/min 3 sessions 标准型 8k tokens/min 10 sessions 增强型 20k tokens/min 25 sessions 专享型 50k tokens/min 60 sessions 注：token_rate 为软限制，超出时排队而非立即拒绝 max_concurrent 为硬限制，达到后立即返回 503

配额超限处理策略

超限类型处理方式用户侧感知
并发会话已满拒绝新会话，返回 503 + retry_after提示"Agent 繁忙，请稍后重试"
Token 速率超限会话内请求排队，最长等待 30s进度条短暂停顿
月度 Token 80%发预警邮件，继续服务管理员收到预警
月度 Token 100%降级至小模型，不停服响应质量轻微下降
会话级 Budget 耗尽完成当前步骤后停止，输出摘要提示"本次任务已达Token上限"

弹性扩容：当业务需求增长时，管理员只需在 M10 系统管理模块中调整 Agent 的 max_concurrent_sessions 和部门 token_monthly_budget，无需重新部署 Agent。新配额在下一次会话准入检查时立即生效。

第5章MULTI-BU INSTANTIATION

多业务单位实例化设计

不同业务单位（部门）的同类业务智能体，是同一 Agent 类型的不同实例。它们共享相同的 Skill 框架，但在技能细节上可能存在微差，知识库与业务数据则完全独立。

定义

配置

验证

审核

发布

第6章CROSS-INSTANCE COLLABORATION

跨实例协作设计

不同业务单位的业务智能体可以通过 A2A 协议互相发起会话、委派任务，实现跨部门协作。协作时，数据隔离原则仍然严格执行：被调用方只返回结果，不暴露原始数据。

A2A 跨实例协作模式

协作模式发起方被调用方典型场景
编排（Orchestrator）审计 Agent财务Agent + 采购Agent年度综合审计报告
流水线（Pipeline）采购 Agent合规 Agent → 法务 Agent合同审查全流程
协商（Consensus）风控 Agent财务Agent + 审计Agent重大风险集体研判

跨实例会话的权限管控

协作时数据隔离：当审计 Agent（AGENT-AUD）通过 A2A 调用财务 Agent（AGENT-FIN）时，财务 Agent 以服务调用方式响应，仅返回分析结论或摘要，不允许审计 Agent 直接访问财务 Agent 的知识库或业务数据。

设计A2A 跨实例会话建立
# 审计 Agent 委派任务给财务 Agent
POST /a2a/AGENT-FIN-01/tasks
{
  "caller_agent_id": "AGENT-AUD-02",
  "task_type": "analyze_expense",
  "input": {
    "period": "2026-Q1",
    "dept_scope": "ALL",
    "anomaly_threshold": 2.5
  },
  // 财务Agent据此决定返回哪些字段
  "output_schema": "anomaly_summary",
  "auth_token": "<A2A_JWT_SIGNED_BY_AUD>"
}
# 财务Agent在独立会话内处理，返回摘要
# 不返回原始账务数据 — 数据不出境

跨实例调用链路

员工发起 ──→ 审计Agent(会话A) │ [A2A 委派, 建立新会话] │ ┌─────────┴─────────┐ ▼ ▼ 财务Agent(会话B) 采购Agent(会话C) [独立执行，访问 [独立执行，访问 财务知识库+数据] 采购知识库+数据] │ │ └─────────┬─────────┘ ▼ [结果汇聚回审计Agent会话A] │ [综合分析 + 生成报告] │ 返回员工 ▶ 全程 TraceID 串联，所有节点记录审计日志 ▶ 财务/采购原始数据不流转至审计Agent

跨实例协作的记忆与经验隔离

即使在 A2A 协作中，各 Agent 的记忆也严格隔离：

记忆类型协作时是否共享说明
企业级记忆✅ 共享跨所有 Agent 的全局知识
部门级记忆（审计部）🚫 不共享仅审计 Agent 自身可读
部门级记忆（财务部）🚫 不共享仅财务 Agent 自身可读
协作经验（跨 Agent）✅ 有条件共享A2A 协作结果摘要可写入企业级记忆（需审批）

第8章API SPECIFICATION

接口规范

业务智能体对外暴露两类接口：面向人类用户/外部系统的 REST/WebSocket 接口，以及面向其他智能体的 A2A 接口。

面向员工 / 外部系统的接口

POST/api/v1/agents/{agent_id}/sessions
创建新会话。返回 session_id 和 WebSocket 连接 URL。若 Agent 已达并发上限，返回 503 + retry_after。

Request：{ "initial_message": "...", "caller_context": {...} }
Response：{ "session_id": "...", "ws_url": "...", "token_budget": 8000 }
WS/v1/ws/sessions/{session_id}
会话主通道，支持流式事件推送。事件类型与 HLD M2 中 StreamEvent 定义一致（delta/thought/tool_call/hitl_required/done）。
GET/api/v1/agents/{agent_id}/sessions/{session_id}/status
查询会话状态（ACTIVE/HITL_WAIT/IDLE/CLOSED）及当前 Token 消耗量。
GET/api/v1/agents/{agent_id}/quota
查询该 Agent 当前并发会话数和配额剩余情况。返回 { current: 7, max: 10, token_remaining_pct: 62 }。

面向其他智能体的 A2A 接口

GET/a2a/{agent_id}/.well-known/agent-card
返回该 Agent 的能力描述卡片（Agent Card），包含支持的任务类型、输入/输出 Schema、自主度级别、所属部门等元数据。
POST/a2a/{agent_id}/tasks
另一个 Agent 向本 Agent 委派子任务，建立 Agent 间会话。

Request：{ "caller_agent_id": "AGENT-AUD-02", "task_type": "...", "input": {...}, "output_schema": "..." }
Response：{ "task_id": "...", "session_id": "...", "status": "accepted" }
GET/a2a/{agent_id}/tasks/{task_id}
轮询 A2A 任务状态，获取执行进度和结果摘要。支持 Webhook 回调替代轮询（在 POST /tasks 时指定 callback_url）。

第9章STATE MACHINE & LIFECYCLE

状态机与生命周期

业务智能体实例状态机

DRAFT → VALIDATING → TESTING → PUBLISHED │ │ │ │ └(校验失败) └(校验通过) └(测试通过) ├→ DEPRECATED → ARCHIVED ↓ ↓ └→ SUSPENDED（违规/紧急停用） PUBLISHED 状态下，接受会话请求： ├── max_sessions 未满 → 建立新会话 └── max_sessions 已满 → 503 + retry_after

单次会话状态机

INITIALIZING（初始化上下文 + 记忆加载） ↓ ACTIVE（ReAct 推理循环中） ├→ HITL_WAIT（等待人工审批，推理暂停） │ └→ ACTIVE（审批通过，恢复推理） ├→ IDLE（推理完成，等待用户下一条消息） │ └→ ACTIVE（用户输入，继续推理） └→ CLOSED ├── 正常：任务完成 / 用户主动关闭 ├── 超时：idle_ttl 到期（默认 30min） ├── 预算耗尽：token_budget 用完 └── 异常：推理引擎故障，强制关闭

CLOSED 后处理：会话进入 CLOSED 状态后，推理引擎释放该会话的 Redis 热缓存（保留 PG 持久记录），并发布 ba.session.closed 事件触发 Mem0 异步记忆提炼，同时从并发计数器中释放一个 slot。
