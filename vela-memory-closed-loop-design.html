Agent记忆闭环系统设计
自我记录 · 自我处理 · 自我检索 · 用户查询与修改

承接此前梳理的25项记忆清单,本设计先筛选出真正需要Agent自动写入、自动使用的部分,构建一个不依赖人工触发的闭环;其余需要人工审批或仅供审计使用的记忆,由独立的"用户查询与修改"通道承载——两条通道共用同一套 MemoryGateway / HitlGate / PermissionChecker / AuditLogger 治理组件。

01范围界定:哪些记忆项进入闭环

判定标准是两个条件同时成立:自动记录(写入不需要人工审批或人工创作)与自动使用(检索/消费由Agent自身或其自动化子系统完成,而非主要面向人工)。只满足其中一个条件的记忆项,不进入闭环,转由第06节的用户查询与修改通道承载。

访问控制决策记录L4✓✗✗供安全团队分析,不参与Agent检索
凭证签发引用记录L4✓✗✗仅供安全审计,不参与Agent检索
HITL复核结论L4✓✗✗供合规报告,不直接参与Agent推理
溯源链记录L4✓✓✓—

三个边界判断的依据

  • 完整交互转录:虽不直接注入prompt,但被自我处理管线自动消费蒸馏,"自动使用"不要求是实时推理输入,自动化内部消费同样算数。
  • 异常与失败事件:兼具面向人工的告警与面向自动聚合(工具能力画像)双重用途,只要后者存在且自动运行,即计入闭环。
  • 组织与业务实体图谱:多数实例走自动提交,少数高敏感属性变更(如关联交易关系)在同一处理流程内部升级HITL——这是风险分级的正常分支,不代表整个记忆项需要被排除。

结果:13/25 项进入自动闭环,其余12项要么写入必须经人工创作/审批,要么消费主要面向人工审计——这12项由第06节的用户查询与修改通道统一承载。

02总体架构:四个子系统

自我记录→自我处理→自我检索构成一个不依赖人工触发的闭环,检索结果又会影响下一轮交互产生新的记录,如此循环;用户查询与修改是与之并行的独立通道,面向全部25项记忆,任何修改都复用同一条 MemoryGateway.write() 治理管线,只是发起方(actor)从Agent换成人类。

03
自我记录

无条件捕获原始交互事件,不做语义判断,对应13项中的L0/L1部分。

04
自我处理

按记忆项类型分派到对应策略,产出候选;提交是否自动取决于风险分级,与处理本身是否自动运行是两回事。

05
自我检索

只读取已提交(自动或经人工批准)的内容,按触发条件决定何时检索,避免每步都全量召回。

06
用户查询与修改

面向全部25项的人工通道,复用同一治理管线,修改已提交事实遵循"旧事实失效、新事实写入"。

03自我记录 Self-Record

自我记录只负责无条件捕获,不判断内容是否重要、是否会被后续使用——这一层不做任何语义决策,只保证"发生过的事情不丢失"。

记忆项触发事件写入模式
任务目标与执行计划ReActEngine完成任务分解同步写入工作记忆状态
会话消息缓冲区每一轮用户/Agent消息产生同步追加
工具调用栈与中间结果MCP/A2A工具调用发起与返回同步写入调用栈
完整交互转录会话/任务结束异步归档(不阻塞主循环)
工具调用日志每次工具调用完成异步落盘
用户反馈/纠错事件用户纠正/点赞点踩/修改输出同步捕获,异步落盘
任务执行轨迹快照每个Thought/Action/Observation节点异步归档(仅供调试,不进入下游处理)
异常与失败事件工具失败/LLM拒绝/超时/策略拒绝同步捕获并同步触发告警,异步落盘

MemoryRecorder(伪代码)

class MemoryRecorder:
    """自我记录:监听运行时事件,无条件写入L0/L1,不做语义判断"""

    def __init__(self, episode_store: EpisodeStore, working_state: WorkingMemoryStore):
        self.episode_store = episode_store
        self.working_state = working_state
        # 事件类型 -> 处理函数 的静态映射;新增记忆项只需注册,不改动分发逻辑
        self._routes = {
            EventType.TASK_PLANNED:     self._to_working("任务目标与执行计划"),
            EventType.MESSAGE_TURN:     self._to_working("会话消息缓冲区"),
            EventType.TOOL_INVOKED:     self._to_working("工具调用栈与中间结果"),
            EventType.SESSION_CLOSED:   self._archive("完整交互转录"),
            EventType.TOOL_COMPLETED:   self._append_log("工具调用日志"),
            EventType.USER_CORRECTED:   self._append_log("用户反馈/纠错事件"),
            EventType.STEP_COMPLETED:   self._append_log("任务执行轨迹快照"),
            EventType.EXCEPTION_RAISED: self._append_log("异常与失败事件"),
        }

    def on_event(self, event: RuntimeEvent) -> None:
        handler = self._routes.get(event.type)
        if handler:
            handler(event)  # 工作记忆同步写入;情景日志异步落盘,不阻塞主循环

04自我处理 Self-Process

处理与提交是两个独立环节:自我处理对所有记忆类型统一自动运行,产出候选;候选是否自动提交还是需要人工复核,由 MemoryGateway.write() 内部的风险分级决定,与"处理这一步是否自动"无关——这正是"反思蒸馏规则""领域知识事实"这类项目虽然处理过程自动、却仍不计入闭环的原因。

原始输入处理策略类型处理逻辑概述输出记忆项提交路径
会话缓冲区+交互转录实体抽取LLM从多轮对话中识别稳定偏好表达用户偏好档案(L2)低风险,自动提交
交互转录+工具日志实体抽取识别新出现的组织/业务实体及关系组织与业务实体图谱(L2)自动提交(高敏感属性个例升级HITL)
完整交互转录任务级摘要任务/阶段结束时对目标里程碑做摘要压缩项目/任务上下文摘要(L2)自动提交
推理链+结论任务级摘要提炼可复用的问题-方案-结论三元组历史结论与经验摘要(L2)自动提交
工具日志+异常事件统计聚合周期性统计各工具在不同条件下的成功率/耗时工具能力与成功模式(L2)自动提交(纯统计,无语义歧义)
交互转录+纠错事件模式蒸馏检测重复失败模式,草拟候选规则反思蒸馏规则(L3草稿)候选自动生成,提交路由HITL
领域文件/专家标注结构化抽取抽取制度条款及适用范围领域知识事实(L2草稿)强制HITL审批(不论来源)
L2事实生成事件指针维护自动记录事实与来源情景的映射关系溯源链记录(L4)自动提交(无实质内容判断)

MemoryProcessor(伪代码)

class MemoryProcessor:
    """自我处理:按记忆项类型分派到对应策略,统一产出候选后交给
    MemoryGateway.write() 做风险分级与提交路由——自动提交还是HITL复核
    由write()内部判定,处理策略本身不关心提交路径,保持关注点分离"""

    def __init__(self, gateway: MemoryGateway):
        self.gateway = gateway
        self._strategies: dict[str, ProcessingStrategy] = {
            "用户偏好档案":       EntityExtractionStrategy(target="user_pref"),
            "组织与业务实体图谱":  EntityExtractionStrategy(target="org_graph"),
            "项目/任务上下文摘要": TaskSummarizationStrategy(),
            "历史结论与经验摘要":  TaskSummarizationStrategy(),
            "工具能力与成功模式":  StatisticalAggregationStrategy(),
            "反思蒸馏规则":       PatternDistillationStrategy(),   # 结果强制走HITL
            "领域知识事实":       EntityExtractionStrategy(target="domain_rule"),  # 结果强制走HITL
            "溯源链记录":         ProvenanceLinkStrategy(),
        }

    def run(self, trigger: ProcessTrigger) -> None:
        # 由MemoryScheduler周期调度,或由SESSION_CLOSED等关键事件触发
        for item_name, strategy in self._strategies.items():
            raw = self._collect_raw_since_last_run(item_name, trigger)
            if not raw:
                continue
            for candidate in strategy.process(raw):
                self.gateway.write(candidate, actor=SYSTEM_PRINCIPAL)

05自我检索 Self-Retrieval

自我检索只面向已经离开活跃上下文窗口的内容——工作记忆(L0)本身就在当前上下文里,不需要"检索"。它按触发条件决定何时查询,而不是每一步都全量召回,避免拖慢主链路延迟。

  • 会话开始时:预加载用户偏好档案与近期项目上下文摘要。
  • 识别到已知实体提及时:检索组织与业务实体图谱、历史结论与经验摘要。
  • 即将调用工具时:检索工具能力与成功模式,辅助工具选型。
  • 输出中带有事实引用时:检索溯源链记录,附加引用来源。

RetrievalTrigger / SelfRetriever(伪代码)

class RetrievalTrigger:
    """决定ReAct循环的哪个节点需要自动触发检索,大多数步骤无需检索,
    直接复用工作记忆即可,以此控制延迟"""

    def plan(self, step: ReasoningStep) -> Optional[RetrievalPlan]:
        if step.is_session_start:
            return RetrievalPlan(items=["用户偏好档案", "项目/任务上下文摘要"], top_k=5)
        if step.mentions_known_entity():
            return RetrievalPlan(items=["组织与业务实体图谱", "历史结论与经验摘要"], top_k=8)
        if step.about_to_invoke_tool():
            return RetrievalPlan(items=["工具能力与成功模式"], top_k=3)
        if step.about_to_cite_facts():
            return RetrievalPlan(items=["溯源链记录"], top_k=len(step.cited_facts))
        return None

class SelfRetriever:
    def maybe_retrieve(self, step: ReasoningStep) -> Optional[MemoryContext]:
        plan = self.trigger.plan(step)
        if plan is None:
            return None
        # gateway.read()已包含RBAC/ABAC前置过滤 + 混合检索 + 可信度重排,此处直接复用
        return self.gateway.read(MemoryQuery.from_plan(plan), actor=step.agent_principal)

06用户查询与修改

这是与自动闭环并行的独立通道,查询范围覆盖全部25项(含仅供审计、需人工审批的12项);任何修改都复用与Agent写入完全相同的 MemoryGateway.write() 管线,只是 actor 换成人类身份,风险分级会据此重新评估。

  • 复核候选:批准/驳回/修改自我处理产出但被路由至HITL的候选(反思蒸馏规则、领域知识事实)。
  • 直接创作:系统提示词、经审核示例库、工作流/SOP模板等本就需要人工撰写的内容。
  • 修正已提交事实:对L2已提交内容(如某条历史结论)提出修改,遵循"旧事实失效、新事实写入"而非物理覆盖。
  • 只读审计查阅:浏览记忆操作审计日志、访问控制决策记录、凭证签发引用记录、HITL复核结论,用于合规检查。

MemoryConsole(伪代码)

class MemoryConsole:
    """用户查询与修改:面向人类的接口层,与Agent自身的自动闭环解耦"""

    def query(self, filters: ConsoleQuery, actor: Principal) -> list[MemoryRecord]:
        # 按人类调用方自身角色鉴权,而非代表Agent的权限集合
        scope = self.permission_checker.resolve_scope(actor)
        return self.store.search(filters, scope=scope)  # 可查草稿/已提交事实/审计日志

    def propose_edit(self, target_id: str, patch: MemoryPatch, actor: Principal) -> WriteResult:
        candidate = MemoryCandidate.from_patch(target_id, patch, writer=actor.id)
        # 与Agent写入路径完全一致:脱敏->冲突检测->策略门->风险分级->双时态提交->审计追加
        return self.gateway.write(candidate, actor=actor)

    def approve_pending(self, proposal_id: str, actor: Principal) -> WriteResult:
        return self.hitl_gate.resolve(proposal_id, outcome=Outcome.APPROVED, reviewer=actor)

07端到端闭环示例

两条路径共用同一套治理组件,区别只在于提交是否需要人工确认。

场景A · 自动闭环

① 用户提到"下午4点后不要安排会议" → ② MemoryRecorder在消息事件中写入会话缓冲区,会话结束后归档进交互转录 → ③ MemoryProcessor的实体抽取策略在下一次调度周期识别出稳定偏好,生成候选 → ④ MemoryGateway.write()判定低风险,自动双时态提交至用户偏好档案,AuditLogger同步追加一条记录 → ⑤ 两周后新会话开始,RetrievalTrigger自动触发检索,该偏好被召回并注入系统prompt,Agent据此主动避开该时段——全程无人工介入。

场景B · 人工修改闭环

① 审计人员在Console中发现某条历史结论关于供应商资质的判断已过时 → ② 通过query()定位到目标记录 → ③ 调用propose_edit()提交修正 → ④ 因目标是已提交的L2事实且涉及资质判断,风险评分判定为中风险,自动路由至HitlGate → ⑤ 另一位有权限的审核人在复核队列中查看该提议连同来源情景与旧结论对比,调用approve_pending()确认 → ⑥ 旧结论被标记valid_to失效(非删除),新结论写入并携带修改人身份,AuditLogger追加一条与原链条哈希相连的修改事件。
