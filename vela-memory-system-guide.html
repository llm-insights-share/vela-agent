01设计前提与四个缺口

LLM 的上下文窗口是有限且无状态的,Agent 要跨会话、跨任务保持连贯性与可进化性,必须依赖外部记忆系统。但主流生产级框架几乎都把优化目标定在"召回准不准、延迟低不低",而不是"这条记忆谁能看、谁批准写入、出了问题能不能查"——这恰恰是 Vela 面向大型国央企私有化部署时无法回避的部分。

综合对 Mem0、Zep/Graphiti、Letta、LangMem、MemOS 五个代表性框架的复核,以及 2025–2026 年新发表的十余篇学术综述,可以确认:目前没有一个主流开源记忆框架把访问控制、凭证隔离、审计留痕和双时态溯源当作原生的一等能力,它们大多假设"单租户、可信环境、只优化召回质量"。这是 Vela 记忆子系统需要自研补齐,而不能直接照搬开源方案的核心原因。本指南据此提出实现方案,并在此基础上给出业界与学术两方面的依据。

缺口 1
RBAC/ABAC 作用域检索

现有框架普遍以 user_id/namespace 做隔离,能防串号,但无法表达"该 Agent 仅在代表某用户执行某工作流时才可访问某数据集"这类组织级、属性级、链路级的权限语义。

缺口 2
Vault/KMS 凭证隔离

会话转录被摘要蒸馏进语义记忆是标准流程,但几乎没有框架在这一步之前内置凭证/密钥的扫描与剥离,存在秘密经摘要"渗出"进长期记忆的风险。

缺口 3
HITL 审计留痕

记忆的增/删/改多为 LLM 自主决策后静默提交,少数框架有异步整理机制,但普遍缺少风险分级的人工复核门与可核验的留痕链条。

缺口 4
双时态溯源

仅 Zep/Graphiti 原生支持双时态(valid time + system time),其余框架多为单一时间戳或干脆物理覆盖旧事实,难以满足审计回放"当时系统知道什么"的要求。

记忆分层与总体架构

在 Mem0/Zep/Letta/LangMem 常见的三层认知记忆分类(情景/语义/程序)之上,Vela 增加第四层"治理记忆",使访问控制与审计成为独立于内容存储的基础设施,而不是散落在各处的附加逻辑。

层级对应概念存储载体关键特性
L0 工作记忆会话内活跃上下文LangGraph Checkpointer(线程级状态)随线程结束"冷却",转录进入 L1 供后续蒸馏
L1 情景记忆原始交互轨迹Postgres / 对象存储,只增不改所有下游记忆的事实源(ground truth episode),携带完整来源指针
L2 语义记忆蒸馏后的实体/事实/偏好Neo4j(双时态图谱)+ Milvus(向量索引)Graphiti 式旧事实失效而非覆盖,支持混合检索
L3 程序记忆固化的技能/规则/Prompt版本化配置存储候选态→生效态必须经 HitlGate 批准
L4 治理记忆不可篡改的操作审计日志独立的哈希链日志存储覆盖 L0–L3 的每一次写/改/失效/删/访问,与内容存储解耦

这一分层直接对应学术界近期综述提出的"写-管-读"(write-manage-read)三阶段循环[1],以及"存储-反思-经验"三段式演化路径[2]——L1 对应存储/写入,后台整理任务对应反思/管理,L2/L3 的检索利用对应经验/读取。区别在于 Vela 在每个阶段都显式插入了策略与审计钩子,而不是把治理当作事后补丁。

三条路径的职责划分

  • 写入路径:候选记忆先过秘密扫描器剥离凭证类文本,再与既有 L2 事实做语义 diff(时态失效而非物理覆盖),经策略门与 HITL 门后双时态提交,最后追加进哈希链审计日志。
  • 管理路径:后台异步的"整理"任务——借鉴 Letta 的 sleep-time agents 与近期 MAGMA 论文的快慢双流设计[3]——完成情景→语义的摘要蒸馏、矛盾检测与衰减策略。"遗忘"是一等公民的可治理操作:默认软删除(标记 invalid_at,保留审计线索),仅在合规强制(如个人信息删除请求)时才触发单独审批的物理清除流程。
  • 读取路径:检索前先解析调用方(用户或代表用户行事的 Agent)的有效权限集合并与记忆的访问控制信封求交集,再执行向量 + 关键词 + 图遍历的混合检索(参考 Zep/Graphiti 的三路融合[4]),最后按新鲜度与可信度重排,抑制可疑或存在冲突的条目被优先召回——这是抵御记忆投毒的第一道防线。

RBAC/ABAC 作用域检索

纯 RBAC 或纯 ABAC 都不足以覆盖多智能体场景:近期研究指出,二者都把每次访问当作独立决策,无法表达"这个 Agent 只有在代表某工作流执行时才能访问某资源"这类跨步骤的委托链条[5]。

  • 每条记忆(不论 L1/L2/L3)携带访问控制信封:owner_id、org_unit、sensitivity_label(对应信创环境常见的密级划分)、allowed_roles、allowed_attributes。
  • 检索请求必须先经过一个独立于记忆存储本身的策略决策点(PDP)解析出调用方的有效权限,再与记忆的访问控制信封求交集——避免策略判断散落进每个存储适配器,这与微软近期提出的"Authorization Fabric"模式一致:PDP 统一评估 RBAC + ABAC + 审批策略,输出 ALLOW / DENY / REQUIRE_APPROVAL / MASK 四种结果[6]。
  • 多智能体协作下,仅有访问控制点还不够,还需要"作用域检索 + 策略化传播"原语,防止 A 智能体为某任务写入的记忆未经允许就悄悄进入 B 智能体的上下文——这正是 2026 年新发表的 MemClaw/ArgusFleet 论文所定义的四个失败模式之一(未授权泄露、陈旧传播、矛盾持久化、溯源断裂)[7]。
  • 默认引入"个人记忆 / 项目记忆 / 组织记忆"三级可见性,对应私有/共享两层模型,并把贡献者、访问资源、时间戳写成不可变的溯源属性,支持事后追溯核验[8]。

凭证隔离(Vault/KMS)

铁律:任何凭证与密钥都不应以明文形式进入记忆存储——理想情况下完全不落入任何一层记忆。

  • Agent 在工具执行沙箱内通过短时任务令牌,按需向 Vault(或 Vela 自建的 CredentialBroker)换取动态凭证;令牌只存活在执行沙箱内,不会进入会后要被摘要蒸馏进记忆的会话转录。
  • 动态、短生命周期凭证是行业公认的实践:每个请求签发唯一凭证,过期自动失效,即使泄露也只在极窄的时间窗口内有效[9]。
  • 在候选记忆进入抽取 LLM 之前,先过一道秘密扫描器——正则规则 + 信息熵检测 + LLM 分类器的多层防御——避免密钥通过摘要"渗出"进语义记忆,这一点在近期"可信 Agentic AI"综述中被明确列为记忆治理的必需环节,而非可选加固[10]。
  • 记忆存储静态加密使用 KMS 包裹的信封密钥,尤其是 L1 原始转录——它离原始工具 I/O 最近,是风险最高的一层。

HITL 审计门与留痕

近期"memorywire"论文把这一模式规范化为 Co-memorize 的"diff-and-approve":Agent 提出记忆变更提案,系统生成结构化差异,人工复核后才提交[11]。

  • Agent 对 L2/L3 提出结构化变更提案(增/改/失效/删)→ HitlGate 按风险打分 → 低风险自动提交,中/高风险进入人工复核队列,附带来源情景、冲突中的既有事实与策略依据 → 审批/驳回/修改本身也是一条被记录在案的审计事件。
  • 风险分级可采用置信度路由:高置信度决策自动执行,中/低置信度升级为人工复核,在自动化收益与人工把控之间取得平衡[12],可复用 LangGraph 的 interrupt() 机制实现暂停-恢复。
  • 审计记录需要能回答"发生了什么、为什么这么做、谁批准的、看到了什么上下文"——只记录事件本身不够,必须包含推理依据与复核结论[13]。
  • 敏感 payload 以哈希/指针形式被引用,而非整段写入日志本体,兼顾可审计性与数据最小化;审计日志本身按留存策略脱敏、限制访问[14]。

审计记录 Schema

event_id · ts · actor(user/agent/system) · operation · target_memory_id · before_hash · after_hash · source_episode_ref · policy_basis · reviewer · rationale —— 只增不改,每条记录内含上一条记录的哈希,形成防篡改链条,满足大型企业内控审计对留痕完整性的要求。

双时态溯源

对齐 Graphiti 的双时态模型[4]:每条语义事实携带两条独立时间线,分别回答"世界上何时为真"与"系统何时知道"。

  • valid_from / valid_to(世界时间):事实在现实中何时成立、何时被取代。
  • created_at / expired_at(系统/摄入时间):记忆何时被写入图谱、何时被标记失效。
  • 被新事实取代的旧事实只失效(设置 valid_to),不删除;物理删除保留给单独的、受法务门控的"清除"流程。
  • 每条事实回溯到产生它的情景记忆,情景记忆回溯到原始事件(工具调用/文档/对话轮次),形成完整的溯源链——这同时满足"当时 Agent 依据什么做出决策"的审计回放需求,与"实体最新状态是什么"的业务查询需求。

BiTemporalFact 数据结构

@dataclass
class BiTemporalFact:
    # 世界时间:事实何时在现实中开始/停止成立
    id: str
    subject: str
    predicate: str
    object: str
    valid_from: datetime
    valid_to: Optional[datetime]      # None = 当前仍然有效

    # 系统时间:摄入与失效的时间戳,非物理删除
    created_at: datetime
    expired_at: Optional[datetime]

    # 溯源与访问控制信封
    source_episode: str               # 指向 L1 原始情景记忆
    writer: str                       # 写入者身份(用户 / Agent)
    sensitivity: SensitivityLabel
    allowed_roles: List[str]
    allowed_attributes: Dict[str, str]

MemoryGateway 核心方法(伪代码)

class MemoryGateway:
    """记忆系统统一入口:所有读写必须经过此网关"""

    def write(self, candidate: MemoryCandidate, actor: Principal) -> WriteResult:
        # 1. 秘密扫描:凭证类文本在进入抽取 LLM 前被剥离
        candidate = self.secret_scanner.redact(candidate)

        # 2. 冲突检测:与既有语义事实做时态 diff(失效而非覆盖)
        conflicts = self.semantic_graph.find_conflicts(candidate)

        # 3. 策略门:ABAC 校验 + 敏感度分级
        policy = self.permission_checker.check_write(candidate, actor)
        if not policy.allowed:
            raise MemoryWriteDenied(policy.reason)

        # 4. 风险分级路由:中/高风险触发人工复核
        risk = self.hitl_gate.score(candidate, conflicts)
        if risk.level in (RiskLevel.MEDIUM, RiskLevel.HIGH):
            decision = self.hitl_gate.request_review(
                diff=candidate.as_diff(conflicts),
                context={"source_episode": candidate.episode_ref,
                         "policy_basis": policy.basis})
            if decision.outcome != Outcome.APPROVED:
                self.audit_logger.append(actor, "write_rejected", candidate, decision)
                return WriteResult(committed=False, reason=decision.rationale)
            candidate = decision.apply_modifications(candidate)

        # 5. 双时态提交 + 向量索引同步
        fact = self.semantic_graph.commit(
            candidate, created_at=now(), writer=actor.id,
            source_episode=candidate.episode_ref)
        self.vector_index.upsert(fact.embedding_view())

        # 6. 审计追加(哈希链,防篡改)
        self.audit_logger.append(
            actor, "write_committed", fact,
            prev_hash=self.audit_logger.tail_hash())
        return WriteResult(committed=True, fact_id=fact.id)

    def read(self, query: MemoryQuery, actor: Principal) -> MemoryContext:
        # 1. 前置作用域过滤:先解析权限,再检索,而非检索后过滤
        scope = self.permission_checker.resolve_scope(actor)

        # 2. 混合检索(向量 + 关键词 + 图遍历)+ RRF 融合
        candidates = self.vector_index.search(query, scope=scope) \
            + self.semantic_graph.traverse(query, scope=scope)
        fused = reciprocal_rank_fusion(candidates)

        # 3. 重排:新鲜度 + 可信度打分,压低可疑/矛盾条目权重
        ranked = trust_aware_rerank(fused, trust=self.compute_trust(fused))

        # 4. 拼装带溯源与时态标注的上下文
        return MemoryContext.assemble(ranked[:query.top_k])

可复用的六个模式

跨越五个框架反复出现、且被多篇 2026 年论文验证有效的工程模式,是比框架本身更值得复用的部分。

热路径与冷路径分离

同步、低延迟的会话内操作与异步、批量的后台整理(摘要蒸馏、冲突检测、衰减)分离——Letta 的 sleep-time agents、MAGMA 的快慢双流、LangMem 的 hot-path/background 都是同一模式的不同实现[3][18]。

混合检索优于单一方法

向量语义检索 + 关键词全文检索 + 图遍历三路融合,再做重排,在时态推理与多跳问题上明显优于纯向量检索——Zep/Graphiti 是这一模式的参考实现[4]。

命名空间隔离是底线而非终点

按 user_id/team_id 做命名空间隔离(LangMem、AWS AgentCore)能防止最基本的串号,但无法覆盖多智能体委托链条,企业场景需要在此之上叠加 ABAC 与作用域传播策略[19]。

溯源即构建,而非事后补充

每条派生记忆在生成时就绑定来源指针(Graphiti 的 episode、Collaborative Memory 的 provenance 属性、MemClaw 的溯源追踪),而不是事后再去反查——2026 年多篇论文将其列为不可再退让的基础设施而非可选加固[7][8]。

遗忘是治理动作

"mnemonic sovereignty"综述明确把"哪些状态可以被遗忘"列为记忆治理需要显式回答的问题之一,而不是缓存淘汰的自然副作用[20]。

基准分数不等于治理就绪

LoCoMo/LongMemEval/BEAM/MemBench 检验的是召回与推理质量,不涉及访问控制或可审计性——框架在这些基准上得分高,不代表满足企业合规要求,这是选型时容易忽略的盲区。

03Academic Frontier · 学术前沿

六个关键动向(2025–2026)

下述动向均来自过去 12 个月内发表或修订的论文,反映学术界正从"记忆能不能召回"转向"记忆能不能被治理"。

趋势 1

分类法开始收敛

三篇独立的 2026 年综述——《Memory for Autonomous LLM Agents》[1]、ACL Findings 2026 的《From Storage to Experience》[2]、OpenReview 的"统一表示-管理视角"综述[21]——分别提出"写-管-读"循环、"存储-反思-经验"三阶段、"自然语言token/中间表示/参数"三范式,彼此高度呼应。同时,Letta 团队的论文重申了一个务实立场:LLM 是"输入token到输出token"的函数而非大脑,过度拟人化的认知类比会误导架构设计,因此采用"消息缓冲区/核心记忆块/归档记忆"这种偏工程实现的分类法可能比照搬情景/语义/程序更利于落地[22]。

趋势 2

2026 成为"记忆变成攻击面"的转折年

OWASP 在 2026 年 Agentic AI Top 10 中新增 ASI06:记忆与上下文投毒,并规定输入审核、带溯源的记忆脱敏、可信度感知检索、行为监控、取证能力五层防御[23]。多项攻击研究显示朴素记忆管线的脆弱性相当显著:

AgentPoison 后门攻击 >80% 成功率 @ <0.1% 投毒率[24]
MINJA 纯查询注入 95%~98% 注入成功率[25]
Agent Security Bench 平均攻击成功率 84.3%[26]

2026 年新发表的长期记忆安全综述[27][20]共同得出一个结论:治理不能在检索或执行阶段"打补丁",必须从存储那一刻起就锚定溯源、版本与留存策略——这与本方案把策略门、HITL 门前置在写入路径中的设计思路完全一致。

趋势 3

多智能体共享记忆治理成为独立子方向

2026 年 6 月发表的 MemClaw/ArgusFleet 论文[7]正式定义了"舰队记忆"(fleet-memory)问题,并给出四个失败模式与对应的系统原语(作用域检索、时态替代、溯源追踪、策略化传播);其生产评测显示,四层深度的溯源链可以 100% 正确重建写入者身份,单跳延迟保持在亚秒级。同期的 Collaborative Memory 论文[8]提出私有/共享双层记忆与二部图访问控制模型,每个记忆片段携带不可变的贡献者与访问记录,支持事后追溯核验。这两篇论文是目前学术界与 Vela 的四个缺口最接近的对应研究。

趋势 4

"良性持久化"风险被单独提出

"mnemonic sovereignty"综述[20]提出一个容易被忽视的观点:记忆系统即便在没有任何攻击者的情况下也会因正常运作而失效——跨用户的静默数据污染、把个人画像事实过度套用到不再适用的场景、记忆引发的谄媚倾向,都属于"良性持久化"风险。这为 Vela 在完全可信的内部部署环境下仍需要冲突检测与策略门(而不仅是面向外部攻击的防御)提供了学术依据。

趋势 5

基准测试仍未覆盖治理维度

LoCoMo[28]、LongMemEval[29]、BEAM、MemBench[30]等主流基准评测的都是信息抽取、多跳推理、知识更新、时态推理等召回与推理能力,没有一个把访问控制正确性或可审计完整性纳入评测维度——这是当前学术界公认但尚未被填补的空白,也意味着"基准分数高"不能作为企业选型的治理证明。

趋势 6

"记忆即操作系统资源"的系统级范式

MemOS[31]提出用 MemCube 统一参数记忆、激活记忆与明文记忆,并通过 MemScheduler、分层机制与治理模块把记忆当作可调度、可审计、可追踪的一等操作系统资源,是目前学术界到生产系统之间架构野心最大的桥梁。但其公开材料的重心仍在跨模态记忆的统一表示与调度上,对企业级多租户 ABAC 着墨有限——这进一步印证了 Vela 要补的四个缺口即便放在最新文献中仍然存在。

04Roadmap & References · 落地路线图

分阶段路线图

按回滚成本从高到低排序:凭证隔离与情景记忆的基础设施一旦补救成本最高,应最先落地;多智能体传播策略依赖前几阶段的信封与审计能力,放在最后。

Phase 1
EpisodeStore + 基础 ABAC 信封 + SecretScanner

先把 L1 只增不改的存储与凭证脱敏做对——这是事后补救成本最高的两块基础设施,后续所有层级都依赖其正确性。

Phase 2
双时态 L2 图谱 + HybridRetriever 融合检索

在 Neo4j 落地 Graphiti 式双时态 schema,接入 Milvus 做语义召回,实现向量 + 图遍历的融合排序。

Phase 3
HitlGate diff-and-approve + 哈希链审计日志

接入风险分级路由与人工复核队列,落地不可篡改的审计记录 schema。

Phase 4
多智能体作用域传播 + 受控遗忘/清除工作流

补齐 MemClaw 式的策略化传播原语,以及合规强制删除的独立审批链路。
