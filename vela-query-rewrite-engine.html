Query改写决策与执行引擎

对话式Agent中,"要不要改写"和"怎么改写"是两个独立问题。本方案给出一套工程可落地的轻量判断机制,以及与之配套的六类改写执行方法,并说明二者如何嵌入现有的检索/工具调用链路。

日期 2026-07-11 适用范围 对话式检索增强 · 工具调用前置处理 · 跨会话记忆检索 状态 设计草案
01 / 方案定位

方案定位与目标

改写不是每轮对话都要执行的重逻辑,而是一个按需触发的旁路能力。它插在"用户输入"和"检索 / 工具调用"之间,只在原始query不足以支撑下游任务时才介入。核心矛盾在于:判断本身必须足够便宜,否则"要不要改写"这道题的成本会超过改写本身带来的收益。

目标一
低误判成本
该改写的漏判会拖累下游命中率,不该改写的误判会引入延迟与语义漂移风险,两类错误的代价并不对称。
目标二
分级可控延迟
判断层必须是毫秒级;只有真正需要时才升级到需要LLM参与的重路径。
目标三
可回退可观测
任何一级改写失败或低置信度,都必须能安全回退到原始query,并留痕供离线分析。
02 / 总览

整体流程总览

三级路由是全案的骨架:分数越高,代表原始query离"可直接使用"越远,允许投入的处理成本也越高。

03 / 判断层

轻量决策机制

判断层的信号必须分层设计:先用近乎零成本的规则信号做粗筛,只有落在"中间地带"的query才值得再花一次embedding或检索前测的成本去精确打分。

3.1 特征信号体系

类型具体特征计算成本示例
规则信号指代词 / 回指命中O(1) 词表查找"它"“这个”“那两个”
规则信号省略句式(缺主语 / 悬空量词 / 疑问词无宾语)O(n) 分词 + 正则"多少钱"“在哪”
规则信号长度过短且无命名实体分词 + NER 查表"换个"(无实体)
规则信号与上一轮实体存在重叠O(1) 缓存比对上轮提及"半导体ETF"
语义信号Query–Context 连贯性缺口1 次 embedding + cosine差值 > 0.3
语义信号检索前测 Top-1 相似度1 次 ANN 查询(按需触发)Top-1 < 0.55
语义信号追问型意图置信度轻量分类器推理置信度 > 0.7

检索前测建议设为条件触发:仅当规则分处于中间地带(既非明显透传也非明显需改写)时才发起一次低成本 ANN 查询,避免每轮都产生额外的检索开销。

3.2 打分与三级路由

规则信号、连贯性缺口、前测惩罚三者加权求和,再按阈值分层路由。伪代码给出可直接映射为工程实现的结构:

Python 
def lightweight_judge(query: str, ctx: DialogueContext) -> RewriteDecision:
    # 1. 规则信号——近似零成本,词表/正则/缓存查找
    rule_score = 0.0
    if has_reference(query, PRONOUN_LEXICON):
        rule_score += 0.4
    if is_elliptical(query):              # 缺主语 / 悬空量词 / 疑问词无宾语
        rule_score += 0.3
    if token_count(query) < MIN_TOKENS and not has_named_entity(query):
        rule_score += 0.2
    if ctx.turn_index > 0 and shares_entity_with_last_turn(query, ctx):
        rule_score += 0.1

    # 2. 语义连贯性缺口——依赖已缓存的句向量,非实时重新编码整段历史
    coherence_gap = semantic_coherence_gap(query, ctx)   # 0~1,越大越需要改写

    # 3. 检索前测——只在中间地带触发,控制额外开销
    probe_penalty = 0.0
    if 0.2 <= rule_score < 0.6:
        top1 = quick_ann_probe(query, top_k=1)
        probe_penalty = max(0.0, RETRIEVAL_CONF_THRESHOLD - top1)

    score = W_RULE * rule_score + W_COH * coherence_gap + W_PROBE * probe_penalty

    if score < TIER0_THRESHOLD:
        return RewriteDecision(tier=0, action="pass_through")
    elif score < TIER1_THRESHOLD:
        return RewriteDecision(tier=1, action="rule_based_rewrite")
    else:
        return RewriteDecision(tier=2, action="llm_rewrite")
分数区间路由动作典型延迟
T0低直接透传原始 query 不变,直接进入下游≈ 0ms
T1中规则改写引擎共指消解 / 拼接实体槽位,不调用 LLM< 10ms
T2高LLM 改写节点HyDE / 多跳分解 / 记忆锚定改写100–300ms

3.3 延迟预算与性能约束

判断层本身不能成为新的瓶颈,建议预算:

  • 规则信号计算:< 2ms
  • 语义连贯性打分(基于缓存句向量):< 5ms
  • 检索前测(仅按需触发):5–15ms
  • 判断层整体目标:P95 < 15–20ms,不应引入用户可感知的等待
04 / 执行层

六类改写方法

六种方法对应不同触发条件,彼此并非顺序关系,而是判断层路由后按场景选用的并列选项。

CRTier 1

共指消解式改写

用最近 N 轮的实体槽位替换代词或省略成分,规则模板或轻量指代链接即可完成,不依赖 LLM。

输入:"它多少钱"
上文实体:A股半导体ETF
输出:"A股半导体ETF多少钱"
EXTier 1

规范化与扩展式改写

同义词 / 领域术语扩展、口语转书面语、拼写纠错。规则词典即可覆盖多数场景;需要更强语义扩展时可升级到 T2 用 LLM 生成变体。

输入:"猪周期咋样了"
输出:"生猪养殖周期反转 最新趋势"
HDTier 2

HyDE 假设文档改写

先用 LLM 生成一段"假设性回答",再用该假设文档的向量去检索,而非直接编码原始 query。适合信息量不足、需要匹配长文档语义的场景。

输入:"玻璃基板供应链影响"
生成假设文档 → 编码 → 检索,而非直接编码原 query
MHTier 2

多跳分解式改写

LLM 将复杂 query 拆解为有依赖关系的子 query 序列(链式或 DAG),配合编排引擎逐步执行、聚合结果。

输入:"对比两家公司的最新财报并给出结论"
→ 子query① 公司A财报 → 子query② 公司B财报 → 子query③ 依赖①②的对比结论
SFTier 1 / 2

结构化槽位改写

把自然语言解析为工具调用所需的参数结构,改写目标从"文本"变为"受 schema 约束的结构化对象"。简单槽位走 T1,涉及推理判断的复杂槽位升级 T2。

输入:"帮我把这条提醒挪到明天下午三点"
输出:{"action":"update","dueDate":"...T15:00"}
MRTier 2

记忆检索式改写

针对"上次讨论的""之前那份方案"这类跨会话模糊引用,补全时间 / 主题锚点,生成适合记忆库检索的具体表达;建议先做一次记忆存在性探测,避免硬造查询。

输入:"上次那份方案发我一下"
输出:"用户上一次讨论的《XX方案》文档"(结合topic标签与时间锚点)
05 / 架构集成

在 Agent Pipeline 中的落点

判断节点与三条改写路径可以直接映射为图编排框架中的条件边与子图,不需要为改写另建一套独立主链路。

组件实现形态典型下游
轻量判断节点条件边(Conditional Edge),读取 state 中的对话历史与实体槽位输出路由标签 T0 / T1 / T2
T1 规则改写引擎独立轻量函数节点,不依赖 LLM 调用直接写回 state.query
T2 LLM 改写节点独立子图,内部可组合 HyDE / 多跳分解 / 记忆锚定写回 state.query 或 state.sub_queries
下游消费方检索器 / 工具调用参数解析器 / 记忆读取接口消费改写后的 query 或子 query 序列

要点:判断节点与改写节点应保持无状态、可单元测试,改写结果作为 state 的显式字段写回,而不是就地修改原始用户输入——这样便于事后回放与效果对比。

06 / 评估

评估指标体系

判断层与改写层需要分开评估,否则很难定位问题出在"该不该改写"还是"改得好不好"。

层级指标说明
判断层触发精确率 Precision判定"需改写"中,真正需要改写的比例
判断层触发召回率 Recall真正需要改写的 query 中被正确识别的比例
判断层决策延迟 P95建议 < 15ms
改写层下游检索提升 ΔRecall@k / ΔNDCG@k改写前后对比
改写层改写引入延迟T1 < 10ms;T2 100–300ms
改写层语义漂移率改写前后 embedding 相似度低于阈值的比例,过高说明改写过度
端到端追问 / 重试率变化改写上线前后对比
端到端工具调用参数解析成功率针对结构化槽位改写路径
07 / 风险

风险与降级策略

改写是一个可以出错的中间步骤,任何环节都需要明确的回退路径。

!
改写置信度低 → 回退原始 query
保留改写候选用于离线分析,但不影响当轮响应,避免"为了改写而改写"。
!
LLM 改写超时 / 异常 → 熔断降级
优先降级到 T1 规则改写,规则改写也失败时直接透传,避免阻塞主链路。
!
过度改写导致语义漂移
对改写前后做一次 embedding 相似度校验,低于阈值则拒绝改写结果并回退。
!
记忆检索改写误召回
先做记忆存在性探测,探测为空时改写为向用户澄清的追问,而非硬造检索式。
08 / 模板

Prompt 模板库

仅 T2 路径需要调用 LLM,以下三份模板对应三种改写目标,均要求模型在无法确定时显式声明,而不是编造内容。

HyDE 假设文档生成

Prompt 
系统:你是检索前置处理器。请针对用户问题生成一段"假设性回答/文档片段"
(不超过150字),尽量覆盖答案可能涉及的实体与术语,但不要声明其为真实答案。

用户问题:{query}
对话上下文摘要:{context_summary}

输出要求:仅输出假设文档正文,不要附加解释。

09 / 附

与记忆系统的衔接建议

Note

本方案的"规则 + 语义 + 阈值分层路由"框架与 MemoryGateway.write() 的风险打分路由是同构设计,建议作为 MemoryGateway.read() / self-retrieval 子系统前置的独立节点复用,而非另起一套判断逻辑——既降低维护成本,也让读写两侧的阈值调参可以共用同一套离线评估数据。

方法⑥"记忆检索式改写"在生成检索锚点前,建议先接入一次记忆存在性探测;探测为空时优先输出澄清追问,避免在记忆库未命中的情况下仍强行编造查询,污染后续 HITL 复核样本。
