01

问题定义:为什么企业内系统需要 Computer Use 能力

大型国企 / 央企内部系统的现实是:OA、ERP、财务、采购、一码通等系统大量缺乏可用 API,或 API 开放程度低、审批链路复杂、系统年代久远。人工在界面上点击操作,是当前唯一稳定、覆盖所有系统的交互方式。

核心价值:把"人工操作界面"变成 Agent 的通用集成手段

Computer Use 类技术的价值在于让 Agent 通过"看屏幕、点鼠标、敲键盘"的方式,像一个新入职员工一样直接使用现有系统界面完成任务 —— 不需要每接入一个系统就单独开发一套 API 集成、申请一次接口权限、等待一轮系统改造排期。这对本来就存在大量"长尾系统"的大型企业尤其有吸引力:一年可能只用到十几次的老系统,值不得做 API 对接,但值得让 Agent 学会怎么点。

现状系统碎片化

央企 / 国企内部往往并存几十上百套系统,新老交替、厂商不一,API 成熟度参差不齐,统一集成成本极高。

现状长尾系统占比高

大量低频但关键的系统(某类审批、某个填报入口)使用量不足以支撑专门的 API 对接投入,长期只能靠人工操作。

诉求可私有化、可治理

不同于面向个人用户的浏览器智能体,企业场景要求私有化部署、强制人工在环、完整审计留痕,不能只靠模型自律。

诉求可复用既有基础设施

不应重新发明凭据管理、风险治理、审计体系,而要接入 Vela 已有的 Integration Gateway、HITL 网关、MemoryGateway 风险打分范式。

边界

驭屏引擎不是要替代更可靠的 API / MCP 集成方式 —— 凡是能走标准 API 或已有 MCP Server 的场景,应当优先走 API。操作界面是"没有 API,或改造成本 / 周期不可接受"时的补充路径与兜底手段,不是默认路径。这一定位贯穿全文的架构取舍。

02

Computer Use / 浏览器智能体的核心实现机制拆解

回答 Q1 · 这类能力的实现机制是什么

不管是 Anthropic 的 Computer Use API、Claude in Chrome,还是各类研究型 Web Agent(WebVoyager、VisualWebArena 等),背后都是同一套"感知 → 定位 → 动作 → 验证"的循环,差异主要体现在感知与定位这两个环节。

01感知(Perception):Agent 怎么"看到"当前界面

Agent 每一步决策前,都需要先获得当前界面状态,常见有两种基础信息源,通常配合使用:

  • 截图(Screenshot) —— 像素级视觉信息,普适性最强:不管背后是网页、桌面客户端,还是十年前的 C/S 架构老系统,只要能截屏就能观测。这也是 Anthropic Computer Use 工具采取的核心方式 —— 模型直接接收截图作为输入来判断下一步动作。
  • DOM / 无障碍树(Accessibility Tree) —— 针对标准 Web 应用,可以拿到结构化的元素树(角色、文本、坐标、可交互状态),信息更精确、噪音更小,但依赖页面本身的实现质量;大量使用 canvas 绘制或无语义 div 堆砌的页面,无障碍树的价值会明显打折。

02定位(Grounding):机制里最关键的一环

模型知道"要点提交按钮"还不够,必须把这个意图转换成可执行的坐标或元素引用 —— 这一步叫 grounding。主流有三种范式:

范式原理优势局限
视觉坐标定位 模型直接从截图预测像素坐标,从屏幕边缘、已知元素等参照点"数像素"确定目标位置 普适性最强,任何 GUI(网页 / 桌面 / 遗留控件)都能用 坐标预测本身有误差,分辨率与缩放变化会带来干扰
标记定位(Set-of-Marks) 程序先在截图上给每个可交互元素画编号框,模型只需在动作里引用编号而非猜坐标 可靠性显著优于纯坐标预测,常与无障碍树文本信息配合使用 依赖能拿到较完整的可交互元素列表,对纯 canvas 渲染效果有限
纯结构化定位 直接用 DOM 选择器 / ARIA 属性定位元素,不依赖视觉 速度最快、成本最低,最易做确定性重放 覆盖不了 canvas 图表、遗留控件等非标准渲染场景

标记定位(Set-of-Marks / SoM)由学术界于 2023 年提出并在 WebVoyager、VisualWebArena 等研究型 Web Agent 中被广泛采用,通常与无障碍树信息配合,兼顾精确性与语义理解 —— 详见第 11 节参考资料。

03动作空间(Action Space):收敛成一套原子操作

不管底层走哪种定位范式,暴露给模型的动作集合通常都收敛成一套很小的原子操作:导航、点击、输入、选择、滚动、等待、上传、截图、提取。这套集合通过标准的工具调用(tool_use)机制交给模型 —— 模型返回结构化的动作请求,执行侧负责真正落地,再把结果(通常是新的截图 / 状态)传回去。这也是 Anthropic Computer Use API 的基本工作方式:模型返回 tool_use 动作,应用在受控的沙箱环境里执行,再把结果作为 tool_result 回传,如此循环往复。

04Agent Loop:观察 → 思考 → 行动 → 验证

循环持续到任务完成或需要人工介入。"验证"环节最容易被低估却至关重要 —— 每次动作后都要确认界面是否真的发生了预期变化(页面跳转了吗?字段真的填进去了吗?),而不是假设动作一定成功;一旦发现异常,要能回退重新观察规划,而不是继续闷头往下走。

05安全机制:护栏思路是企业化设计的直接依据

这类能力天生高风险,因为 Agent"看到什么就可能被什么误导"。Anthropic 在 Claude for Chrome 上验证过几条很值得借鉴的护栏设计:按站点 / 域名做权限颗粒度控制;对"发布、购买、分享个人数据"这类高风险动作,即便用户开启了自主模式也仍保留强制确认;并专门训练了针对提示词注入的分类器 —— 检测到疑似注入指令时,会引导模型停下来向用户确认而非直接执行。公开的红队测试数据显示,这类专项防护能把浏览器特定攻击的成功率从 35.7% 降到 0%,整体提示词注入攻击成功率从 23.6% 降到 11.2%。

启示

11.2% 仍意味着约九分之一的攻击可能得手 —— 这说明"模型自律 + 用户临场确认"在企业无人值守场景里远远不够。第 05、08 节里,驭屏引擎把"关键动作强制确认"做成不可关闭的硬约束,而非可选项,正是对这一点的直接回应。

03

企业内系统场景 vs 开放 Web 场景的关键差异

通用 Computer Use / 浏览器智能体面向的是开放互联网、有人实时在场监督的场景。企业内系统操作的假设完全不同 —— 这些差异直接决定了驭屏引擎不能照搬通用方案,而要做针对性的架构取舍。

维度开放互联网场景企业内系统场景(Vela 需应对)
部署边界可调用云端 API / 云端模型必须私有化部署,截图 / DOM / 模型推理均不出内网
系统形态以现代 B/S 网页为主大量遗留 C/S 桌面客户端与老旧 B/S 系统并存
定位范式优先级视觉坐标定位为主(通用性优先)DOM / 无障碍树 + SoM 优先(更快、更省、更可审计),视觉兜底遗留控件
凭据处理用户当场手动登录 / 授权必须走 Vault / KMS 托管代理,凭据不进入模型上下文
执行频率多为一次性 / 低频任务大量周期性重复任务(报销、对账、审批查询)→ 需要"编译固化 + 确定性重放"
风险容忍度用户实时在场,可随时打断无人值守场景居多,必须靠强制网关兜底,而非临场监督
审计要求一般无强制留痕要求需要完整、不可篡改的操作留痕,对齐企业审计 / 合规要求
注入攻击暴露面面对整个公网,暴露面极大相对封闭的内网环境,但一旦发生,影响涉及真实业务数据

05

关键机制详解

逐层展开实现要点,并以"提交一张差旅报销单"为例做端到端案例走查。

L1 · PER感知层:双通道采集

  • 浏览器模式下,通过 Playwright 的无障碍树快照 API 拿到结构化元素信息,配合 CDP 截屏,两者按时间戳对齐,减少"看到的"和"能点的"不一致。
  • 桌面模式(遗留 C/S 客户端)下无 DOM 可用,退化为纯截图观测。

L2 · GND定位层:SoM 标注 + 选择器指纹库

  • SoM 标注器:遍历无障碍树中可交互节点(role 为 button / link / textbox / combobox 等),在对应截图区域画编号框,生成"标注截图 + 编号↔元素映射表",提供给 RSN 层引用。
  • 选择器指纹库:为每个元素同时记录多种候选定位方式(id、稳定 CSS 路径、XPath、ARIA label、相对视觉位置),供 L5 技能重放时"自愈" —— 某一种定位方式失效时按优先级尝试下一种,而不是整任务失败。
  • 桌面模式兜底:无法获得无障碍树时(遗留控件、canvas 看板),回落到视觉坐标定位,与 Anthropic 原生 Computer Use 工具"从参照点数像素"的方式一致。

L3 · ACT动作执行层:双执行后端

  • 浏览器模式:Playwright 驱动池,每个任务分配独立、隔离的浏览器上下文(不共享 cookie / 缓存),网络出口限定在系统注册表登记的域名白名单内。
  • 桌面模式:虚拟显示容器 + 坐标级鼠标键盘合成,覆盖浏览器触达不到的老旧 C/S 客户端(部分财务 / 一码通桌面端)。
  • 每个动作执行后做验证:比较动作前后的 DOM / 截图差异或等待网络空闲,确认状态确实发生了预期变化,而非默认动作已成功。

L4 · RSN推理决策层:LangGraph 子图

Observe–Plan–Act–Verify 循环用一个 LangGraph 子图表达,失败时回退重新观察规划,连续失败则升级人工:

from langgraph.graph import StateGraph, END

def observe_node(state):
    # PER + GND:截图 + 无障碍树 + SoM 标注
    state["obs"] = perceive(state["session"])
    return state

def plan_node(state):
    # RSN:调用 Qwen / DeepSeek 规划下一步动作
    state["action"] = llm_plan(state["goal"], state["obs"], state["history"])
    return state

def act_node(state):
    # ACT:Playwright 驱动池 / 桌面控制通道执行,GOV 网关在此拦截 T2/T3 动作
    state["result"] = execute(state["action"])
    return state

def verify_node(state):
    state["ok"] = verify(state["result"], state["action"])
    return state

def route(state):
    if state["ok"] and state["done"]:
        return END
    if not state["ok"] and state["retries"] < 3:
        return "observe"
    if not state["ok"]:
        return "escalate_human"  # 超过重试上限,升级人工而非继续尝试
    return "observe"

graph = StateGraph(dict)
graph.add_node("observe", observe_node)
graph.add_node("plan", plan_node)
graph.add_node("act", act_node)
graph.add_node("verify", verify_node)
graph.add_conditional_edges("verify", route)

L5 · SKL技能编译与缓存层:探索一次,确定性重放

企业里大量任务是周期性重复的(周报销、月对账)。若每次都全程 LLM 逐步推理,成本高、结果有波动、也难以向审计解释"为什么这次点了不同的按钮"。因此:

  • 首次执行走完整 Agent Loop,同时把每一步(动作类型、目标元素指纹、输入值来源、验证结果)记录为轨迹。
  • 技能编译把轨迹抽象成参数化模板(具体输入值替换为参数占位符),存入技能库 —— Neo4j 存"任务 – 步骤 – 系统"关系图,Milvus 存任务描述向量,供后续按语义相似度检索匹配。
  • 确定性重放后续同类任务直接按模板执行,不逐步过 LLM;只在某一步全部候选定位方式都失效时,才触发自愈 —— 升级回 L4 做局部重新规划,修复成功后更新技能模板。
对照

这一思路与开源项目 Skyvern 的做法接近:先在真实浏览器里跑一遍并记录动作,生成可缓存复用的工作流,界面变化时靠 AI 能力自愈而无需人工重写脚本。区别在于,驭屏引擎需要在私有化环境内闭环完成整个编译 / 重放过程,并把它纳入 GOV 层的审计留痕。

L6 · CRD凭据与会话代理层:复用 Integration Gateway

  • 直接复用既有 Integration Gateway 的 OAuth 2.1、RFC 9728/8707 令牌绑定、Vault / KMS 托管体系,不重新发明凭据管理。
  • 新增表单登录宏:遇到不支持 OAuth 的老系统,预置一段确定性的、非 LLM 驱动的登录脚本,由 GND 层固定选择器执行 —— 登录环节风险太高,不适合交给推理型 Agent 临场决定。
  • 会话生命周期:任务开始时从 Vault 取凭据、建立会话;任务结束或超时后主动销毁会话,不做凭据持久化。

L7 · GOV治理与审计层:强制网关,而非可选项

  • 风险打分:参照 MemoryGateway 已有的风险打分范式,对每个即将执行的动作按"读 / 写 / 关键"三级打分(详见第 08 节矩阵)。
  • 强制网关:关键动作(提交、删除、审批通过、资金相关)必须命中 HITL 网关,展示预览 / diff 后才能继续 —— 且这条策略不受 Agent 自主模式开关影响,呼应第 02 节里 Claude for Chrome"即使自主模式也保留高风险确认"的护栏思路,但在企业场景里做成不可关闭的硬约束。
  • 审计留痕:每一步截图 / DOM 快照 / 动作 / 校验结果 / 人工确认记录写入既有仅追加审计日志,支持双时态溯源,复用 MemoryGateway 已确立的 provenance 模式。

案例走查:在 OA 系统中提交一张差旅报销单

1
任务下发 RSN
Agent 收到"帮我提交上周的差旅报销单",RSN 层将其分解为登录 → 定位表单 → 填写 → 提交等子目标。
2
凭据代理登录 CRD
从 Vault 取 OA 系统凭据,建立隔离浏览器会话,走预置登录宏完成登录 —— 由固定脚本执行,非 LLM 临场决策。
3
感知与标注 PER · GND
观测"报销申请"页面,SoM 标注出"费用类型""金额""附件上传""提交审批"等元素及编号。
4
逐字段填写与校验 RSN · ACT
依次选择费用类型、填写金额、上传附件、填写摘要;每步后 VERIFY 确认字段已生效再进入下一步。
5
命中强制网关 T3 · GOV
到达"提交审批"按钮,GOV 层识别为关键动作,暂停执行,把表单摘要推送给人工确认。
6
人工确认后提交 ACT
人工确认后执行提交,VERIFY 确认页面出现新的单据编号,判定任务成功。
7
技能固化 SKL
整个轨迹编译为"OA 差旅报销提交"技能模板,供下次同类任务确定性重放。
8
审计留痕 GOV
写入完整审计记录:发起人、执行 Agent、确认人、时间戳、前后截图,支持事后追溯。
06

与 Vela 既有架构的集成关系

驭屏引擎不是孤立新增的模块,而是最大程度复用已有基础设施 —— 这也是它能较快落地、且天然继承既有治理水位的原因。

已有 Vela 模块被哪一层复用具体复用内容
Integration GatewayL6 · CRDOAuth 2.1、RFC 9728/8707 令牌绑定、Vault / KMS 凭据托管体系
HITL 网关状态机L7 · GOVgate-check 状态机与强制确认流程
MemoryGateway 风险打分范式L7 · GOV动作级风险打分方法论与阈值路由逻辑
Milvus / Neo4jL5 · SKL技能库的语义向量检索(Milvus)与任务 – 步骤 – 系统关系图谱(Neo4j)
MCP / A2A 协议层接口暴露层与既有 Agent 工具调用体系保持一致的对外接入方式

换句话说:驭屏引擎新增的核心是 PER / GND / ACT / RSN / SKL 这套"看屏幕、点鼠标"的能力本身;凭据怎么管、风险怎么控、审计怎么留,直接沿用平台已经验证过的成熟范式,不重新发明。

07

接入指南:如何为你的 Agent 增加这个能力

回答 Q2 · 如何为已有 Agent 增加此能力

对已有的 Vela LangGraph Agent 而言,接入驭屏引擎是一次标准的 MCP 工具接入,不需要改动 Agent 自身的编排逻辑。

1
注册目标系统
在系统注册表登记入口地址、登录方式(表单 / SSO / CAS)、执行模式(浏览器 / 桌面)。
2
选择执行后端
B/S 系统默认走浏览器模式;确认为遗留 C/S 桌面客户端时启用桌面控制通道。
3
接入凭据代理
通过 Integration Gateway 把目标系统凭据托管进 Vault;不支持 OAuth 的老系统需录制一次登录宏。
4
标注风险分级策略
参照第 08 节矩阵,为该系统的关键动作打 T0–T3 标签,明确哪些必须命中强制确认。
5
注册为 MCP 工具并启用
在 Agent 的工具清单里加入 ui_navigate / ui_observe / ui_act / ui_extract / ui_replay_skill,像调用其他 MCP 工具一样调用即可。
// mcp.config.json —— 将驭屏引擎注册为 MCP Server
{
  "mcpServers": {
    "vela-screenpilot": {
      "url": "https://screenpilot.vela.internal/mcp",
      "tools": ["ui_navigate","ui_observe","ui_act","ui_extract","ui_replay_skill"]
    }
  }
}
# 已有 Agent 侧几乎零改动 —— 追加工具即可
screenpilot_tools = load_mcp_tools(server="vela-screenpilot")
agent = create_agent(
    model=qwen_max,
    tools=[*existing_tools, *screenpilot_tools],
)
08

风险分级矩阵与治理策略

GOV 层据此矩阵对每个待执行动作打分,决定是自动放行、抽样复核,还是强制暂停等待人工确认。

等级定义示例动作处理策略
T0只读观测,不改变任何系统状态navigate / screenshot / extract自动执行,仅记录日志
T1改变页面 / 表单本地状态,尚未提交type / select / scroll / 暂存上传自动执行 + 事后抽样复核
T2触发系统侧持久化状态变更点击"提交""保存""发送"暂停,展示预览 / diff,人工确认后执行
T3涉及资金、审批、权限、删除的关键操作审批通过、支付、删除记录、权限变更强制双人复核,路由进企业既有审批流,Agent 不自主决定
硬约束

T2/T3 的强制确认不受任何"自主模式"配置项影响 —— 这是驭屏引擎与消费级浏览器智能体最本质的差异:后者把风险控制做成用户可关闭的偏好,前者把它做成系统不可关闭的边界。

落地路线图

P0 · 当前季度
  • 浏览器模式 MVP,覆盖 3–5 个高频 OA / 费控场景的只读查询与表单填写
  • 不含自动提交,由人工完成最终点击确认
  • 打通 CRD 层与 Integration Gateway 的凭据托管
P1 · 下一阶段
  • SKL 层上线,支持技能编译与确定性自愈重放
  • 桌面模式(C/S 客户端)接入 POC 验证
  • GOV 层风险分级规则化,审计日志接入既有留痕体系
P2 · 远期
  • 风险分级策略自动化优化,减少不必要的人工打断
  • T3 动作直接路由进企业既有审批流,而非仅弹窗确认
  • 跨系统多步编排(如 OA 请假审批通过后自动同步至考勤系统)
  • 技能模板纳入 Vela 技能商店,支持跨项目复用发布

驭屏引擎在机制上吸收了 Set-of-Marks 定位、Claude for Chrome 的护栏思路、Skyvern 式"探索一次 · 编译固化 · 自愈重放"模式,但企业化落地上做了显著增强。
